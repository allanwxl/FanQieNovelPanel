# 番茄短故事数据增强面板｜后续功能规划

> 文档版本：v1.0｜更新日期：2026-06-26

---

## 一、背景与目标

### 1.1 现状

当前面板已完成 MVP 阶段：

- 复用番茄作家后台登录态，只读拉取作者全部短故事作品数据
- 本地 IndexedDB 存储历史指标，支持排序、筛选、趋势图、导出
- 爆款潜力评分与推广标记功能已上线

### 1.2 痛点

作者日常运营中仍存在大量重复手工操作：

| 痛点 | 现状操作 | 期望能力 |
| --- | --- | --- |
| 小说章节上传 | 在番茄作家后台逐章手动粘贴、提交审核 | 批量导入章节草稿，一键提交 |
| 短故事上传 | 在官方后台逐篇创建、编辑、发布 | 批量上传短故事，模板化编辑 |
| 微头条推广 | 手动复制作品链接、撰写文案、发布微头条 | 从面板直接发布微头条推广内容 |

### 1.3 目标

在现有数据增强面板基础上，逐步引入**内容生产能力**，从"看数据"升级为"看数据 + 做运营"，形成完整的短故事运营闭环。

## 五、技术实现思路

### 5.1 小说章节批量上传

#### 5.1.1 文件解析模块

```text
src/
  upload/
    parsers/
      txtParser.ts       # TXT 文件按章节分割
      docxParser.ts      # DOCX 解析（需引入 mammoth 或类似库）
      markdownParser.ts  # Markdown 按标题分割
      index.ts           # 统一解析入口
```

核心逻辑：

- TXT：按正则 `第[一二三四五六七八九十百千\d]+章` 或空行分割
- DOCX：使用 `mammoth` 库提取纯文本后按标题分割
- Markdown：按 `##` / `###` 标题层级自动分割

#### 5.1.2 章节管理 UI

```text
src/ui/pages/
  ChapterUpload.tsx     # 章节上传页面
  ChapterEditor.tsx     # 章节内容编辑器
```

功能要点：

- 拖拽上传区域 + 文件选择按钮
- 章节列表展示（可拖拽排序）
- 单章编辑弹窗（标题 + 内容 + 字数统计）
- 批量操作栏（全选、删除、提交审核）

#### 5.1.3 上传策略

- 逐章提交，间隔 500ms-1000ms，避免触发频率限制
- 上传失败的章节标记为"失败"，支持单独重试
- 上传进度实时展示（已提交 / 成功 / 失败 / 待处理）

### 5.2 短故事上传

#### 5.2.1 内容编辑器

```text
src/ui/components/
  StoryEditor.tsx        # 富文本编辑器
  CoverSelector.tsx      # 封面选择器
```

技术选型：

- 编辑器：使用 `@tiptap/core` 或 `prosemirror` 构建轻量富文本编辑器
- 或简化方案：直接使用 `textarea` + Markdown 预览

功能要点：

- 标题输入（必填，限 30 字）
- 正文编辑（支持富文本或 Markdown）
- 字数统计（实时显示）
- 标签选择（从预设标签中选择）
- 封面设置（可选）

#### 5.2.2 批量上传方案

```text
src/
  upload/
    batchStoryUpload.ts   # 批量上传编排
```

- 支持导入 JSON/CSV 格式的多篇故事元数据
- 每篇故事独立上传，失败不影响其他
- 上传队列管理，支持暂停/恢复

### 5.3 短故事发布微头条

#### 5.3.1 推广发布模块

```text
src/
  publish/
    weitoutiaoPublish.ts  # 微头条发布逻辑
   文案Templates.ts       # 推广文案模板
```

#### 5.3.2 文案模板系统

预设模板示例：

```text
📖 {title}

{excerpt}

📊 数据表现：
· 阅读量：{readers}
· 触底率：{completionRate}
· 点击率：{clickRate}

#短故事 #番茄小说
```

功能要点：

- 模板变量自动填充（从作品数据中提取）
- 支持自定义模板
- AI 辅助生成文案（可选，调用大模型 API）

#### 5.3.3 发布流程

1. 从数据榜单选择作品 → 自动生成推广文案
2. 用户编辑文案 → 预览微头条卡片效果
3. 选择发布时间（立即/定时）→ 调用发布接口
4. 发布成功 → 自动更新推广标记 + 记录发布历史

## 六、风险与依赖

### 6.1 技术风险

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 官方无批量上传接口 | 只能逐章/逐篇提交，效率受限 | 封装自动重试 + 进度追踪，优化用户体验 |
| 写入接口需要额外鉴权 | 可能需要 CSRF Token 或其他安全参数 | 通过 Content Script 注入页面上下文获取 |
| 富文本编辑器复杂度高 | 开发周期长，维护成本大 | MVP 阶段使用纯文本 + Markdown |
| 微头条发布接口不稳定 | 发布失败率高 | 本地暂存草稿，失败后可重试 |
| 官方接口字段变化 | 已有字段映射失效 | 保留 raw 响应，标准化层集中适配 |

### 6.2 合规风险

| 风险 | 应对 |
| --- | --- |
| 写入操作可能违反平台 TOS | 明确标注"仅限个人使用"，不提供商业化服务 |
| 高频请求可能触发风控 | 严格控制请求间隔，添加随机延迟 |
| 内容安全审核 | 接入敏感词检测，发布前提示用户确认 |

### 6.3 外部依赖

| 依赖 | 用途 | 是否必需 |
| --- | --- | --- |
| mammoth | DOCX 文件解析 | 可选（V2 阶段） |
| @tiptap/core | 富文本编辑器 | 可选（可用 textarea 替代） |
| 大模型 API | AI 生成推广文案 | 可选（V3 阶段） |

| 任务 | 估时 | 依赖 |

## 十、总结

本规划围绕三大核心能力展开：

1. **短故事发布微头条** — 复用数据榜单，最小改动实现运营闭环
2. **短故事上传** — 补充内容生产能力，结构简单易实现
3. **小说章节批量上传** — 技术复杂度最高，建议最后实施

建议按 Phase 1 → Phase 2 → Phase 3 的顺序推进，每个阶段独立交付、独立验收，降低整体风险。

关键前置条件：**接口调研必须先行**。在确认官方写入接口的可用性和参数结构后，才能进入实质开发阶段。
